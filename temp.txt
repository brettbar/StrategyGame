Great question! In a GOAP (Goal-Oriented Action Planning) system, **accruing effects** — like building multiple settlements to reach `HasSettlements=3` — require special handling because standard GOAP planners typically treat effects as binary or fixed-value assignments, not additive or cumulative.

Here’s how you can extend your GOAP planner to handle **accruing effects**:

---

## ✅ 1. Represent World State with Numeric Values

Instead of just booleans, allow your `WorldState` to hold numeric values:

```cpp
struct WorldState {
    std::unordered_map<std::string, int> intValues;
    std::unordered_map<std::string, bool> boolValues;

    int getInt(const std::string& key, int defaultValue = 0) const {
        auto it = intValues.find(key);
        return it != intValues.end() ? it->second : defaultValue;
    }

    void setInt(const std::string& key, int value) {
        intValues[key] = value;
    }

    // ... similar for bools
};
```

---

## ✅ 2. Extend Action Effects to Support Delta Changes

Instead of only setting absolute values, allow effects to specify **relative changes** (deltas):

```cpp
struct GoapAction {
    std::string name;
    std::unordered_map<std::string, int> preconditions;  // e.g., "Wood >= 5"
    std::unordered_map<std::string, int> effectsDelta;   // e.g., "HasSettlements += 1"
    std::unordered_map<std::string, int> effectsSet;     // e.g., "IsBuilding = 1" (absolute)

    int cost = 1;

    bool CheckProceduralPrecondition(const WorldState& state) const {
        for (const auto& [key, required] : preconditions) {
            if (state.getInt(key) < required) return false;
        }
        return true;
    }
};
```

> 💡 You can also support both `effectsSet` (absolute assignment) and `effectsDelta` (relative change) — or unify them with a flag or enum.

---

## ✅ 3. Modify the Planner’s State Transition

When applying an action’s effect, apply both delta and set effects:

```cpp
WorldState ApplyAction(const WorldState& state, const GoapAction& action) {
    WorldState newState = state;

    // Apply delta effects
    for (const auto& [key, delta] : action.effectsDelta) {
        int current = newState.getInt(key);
        newState.setInt(key, current + delta);
    }

    // Apply set effects
    for (const auto& [key, value] : action.effectsSet) {
        newState.setInt(key, value);
    }

    return newState;
}
```

---

## ✅ 4. Handle Goals That Require Numeric Thresholds

Your goal condition should check if a value meets or exceeds a target:

```cpp
struct GoapGoal {
    std::string name;
    std::unordered_map<std::string, int> requiredState; // e.g., "HasSettlements >= 3"

    bool IsFulfilled(const WorldState& state) const {
        for (const auto& [key, target] : requiredState) {
            if (state.getInt(key) < target) {
                return false;
            }
        }
        return true;
    }
};
```

---

## ✅ 5. Planner Must Allow Repeated Actions (If Needed)

Standard GOAP planners often avoid cycles or repeated actions. But for accruing effects, you may need to allow the **same action to be used multiple times** — as long as preconditions still hold.

You can handle this in two ways:

### Option A: Let the planner naturally repeat the action

If your search algorithm (like A* or Dijkstra) doesn’t forbid revisiting states, and the action’s cost is positive, it will eventually stop when the goal is met.

> ⚠️ But beware of infinite loops — always bound your search depth or use visited state caching with cost comparison.

### Option B: Generate “multi-action” variants dynamically

You could dynamically generate actions like:

- `BuildSettlement x1`
- `BuildSettlement x2`
- `BuildSettlement x3`

...each with multiplied costs and effects. But this is messy and doesn’t scale.

### ✅ Recommended: Just allow action repetition with state caching

In your planner’s search (e.g., A*):

```cpp
struct SearchNode {
    WorldState state;
    std::vector<const GoapAction*> actionPath;
    int cost = 0;

    // For A* priority_queue
    int estimatedTotalCost = 0;
};

// In your search loop:
for (const auto& action : allActions) {
    if (!action.CheckProceduralPrecondition(currentNode.state)) continue;

    WorldState nextState = ApplyAction(currentNode.state, action);
    int newCost = currentNode.cost + action.cost;

    // Check if we’ve seen this state with lower or equal cost → skip if so
    if (visited.count(nextState) && visited[nextState] <= newCost) continue;

    visited[nextState] = newCost;

    // Create new node and push to queue...
}
```

> You’ll need to define `operator==` and `std::hash` for `WorldState` to use it in `std::unordered_map`.

## ✅ 6. Example: BuildSettlement Action

GoapAction buildSettlement;

Goal:

The planner will now (if resources allow) chain `BuildSettlement` → `BuildSettlement` → `BuildSettlement` to meet the goal.

## ✅ Summary

To handle accruing effects like `HasSettlements=3`:

- Use **numeric world state values**.
- Support **delta effects** (`+=`) in actions.
- Allow **repetition of actions** in planning.
- Define goals as **numeric thresholds**.
- Use **cost-aware search** (A*/Dijkstra) with **state caching** to avoid redundant work.

---

## 🧠 Advanced Tip: Resource-Driven Accrual

If you want to get fancy, you can even plan *how many times* to repeat an action based on available resources:

> “I have 30 Wood → I can afford 3 BuildSettlement actions → that gets me to goal.”

You could precompute this and inject a “BatchBuildSettlement x3” action dynamically — but letting the planner discover it naturally is often simpler and more flexible.
